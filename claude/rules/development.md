# 開発ルール

## ブラウザ操作ツールの優先順位

- **Chrome DevTools MCP（`mcp__chrome-devtools__*`）を最優先で使う**。AI エージェント専用 Chrome profile（`chrome-ai-agent` / `[::1]:9222`）に接続する
- **Claude in Chrome MCP（`mcp__Claude_in_Chrome__*`）** は Claude Code 拡張が管理する別ツール。Chrome DevTools MCP とは独立して使える
- AgentCore Browser はローカル開発では使わない（AgentCore Runtime 上のエージェント用）
- Chrome DevTools MCP のセットアップ手順・トラブルは `/kb-ts-claude-code` を参照

### AI エージェント専用 Chrome profile

- 起動: `chrome-ai-agent`（`~/.local/bin/chrome-ai-agent`）／user data dir: `~/.config/chrome-ai-agent`／remote debugging: `[::1]:9222`
- 1Password 拡張はこの profile に入れ、みのるんがログイン部分だけ操作する
- **通常利用の Chrome profile をリモートデバッグ用途へ流用しない**
- **例外**: みのるんが明示的に「既存ブラウザセッションを渡す」と指示した場合だけ一時的にその endpoint へ接続し、作業後は `chrome-ai-agent` / `[::1]:9222` に戻す

## 基本

- コンテキスト節約のため、調査やデバッグにはサブエージェントを活用する
- 開発中に生成するドキュメントに API キー等の機密情報を書く場合は、必ず .gitignore に追加する
- 「変更不要」「問題なし」と結論する前に、必ずその根拠を明示する。ドキュメントやソースを確認せずに「正確です」と言わない
- **全体スコープへの断定**（「このコードベースには〜がない」等）は、必ず `grep -rn` で全ファイル横断スキャンしてから言い切る。一部のファイルだけ見て全体を断定しない
- 判定一覧を出すときは、各行の**根拠の信頼度（実コード確認 / ドキュメント確認 / 主観 / 推測）**を明示する
- パッケージ・ライブラリを選定するときは、選んだ理由と代替案を必ず示す
- WebFetch で URL 本文が取得できない場合（JS 描画ページ等）は、推測で進めずみのるんに本文のコピペを依頼する
- Docker が必要な作業の前に、Docker が起動していなければ `startdocker` で自分で起動する（みのるんに依頼しない）

## 大規模出力を親コンテキストに取り込まない（`Prompt is too long` 対策）

`git diff` 生パッチ・大量 grep 結果・PDF 全文・長大ログ等を親のメッセージ履歴に直接流し込まない。親の膨張は後続ツール呼び出しやサブエージェント起動プロンプトに波及し、MCP 定義が重い環境では特に上限超過しやすい。

| 出力の目安 | 処理方針 |
|-----------|---------|
| 500 行未満 / 数十KB 以下 | 親で直接読んで OK |
| 500 行以上 / 数百KB 以上 / 数十ファイル分の diff / PDF 30ページ超 | サブエージェント（general-purpose）に委譲し、要約・判定結果（表・JSON）だけ受け取る |

サブエージェント委譲の作法：

- プロンプトは**自己完結**で書く（「親のコンテキストを参照」と書かず、必要な情報・ルールを再掲する）
- 戻り値はフォーマット固定（JSON / 表）にして生データを返させない。並列時は戻り値の合計サイズも指定する（例：上位N件・1件200字以内）
- 編集・コミット等の副作用のある操作は親が担当し、子は読み取り・分析に限定する

## サブエージェント並列起動は原則逐次

「MCP 定義が重い環境 × 親コンテキストの膨張 × 複数並列」が揃うと `Prompt is too long` で即失敗する。

対策の優先順：①並列をやめて逐次処理（完走確実性優先）②親のコンテキストを事前に軽くする ③子に渡す prompt を簡潔にする ④どうしても並列なら 2〜3 個まで。

スキルで Agent 並列起動を設計するときは、SKILL.md に「並列化できる根拠」を明記する。根拠がなければデフォルトで逐次。

## jq の `\(...)` 文字列補間と bash の衝突

jq の文字列補間 `"\(.field)"` を**二重引用符**の jq フィルタに入れると、bash が `\` を消費して `syntax error, unexpected INVALID_CHARACTER` になる。

- **jq フィルタはシングルクォートで囲む**：`jq -r '"- name: \(.name)"' "$f"`
- フィルタ内で bash 変数が必要ならヒアドキュメントで渡す
- 複数の補間や複雑な join は最初から Python に切り替える（シェルクォート地獄を回避）

## 複数作業の混線防止（CWD 干渉・ステージ範囲）

- Bash の並列実行間でも **working directory は共有される**。片方の `cd` がもう片方の git 操作先を変えてしまう事故が起きうる。`cd` を使わず **`git -C <path>`** でリポを明示し、異なるリポの更新を同一メッセージで並列にしない
- `git add -A` / `git add <dir>/` は**並行セッションの未コミットファイルを巻き込む**。対策の優先順：
  1. 自分が触ったファイルパスを明示列挙して add する
  2. `git commit <path> -m "..."` でパス指定コミット（最も確実）
  3. コミット前に `git status --short <dir>/` で目視。見覚えのないファイルは別セッションの作業
  4. `git add -A` / `git add <dir>/` は「自分しか触っていない」と確信できるときだけ
