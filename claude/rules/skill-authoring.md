# スキル作成・更新ルール（全プロジェクト共通）

`.claude/skills/<name>/SKILL.md` を新規作成または編集するときの共通ルール。ユーザースコープでもプロジェクトスコープでも同じ方針を適用する。

## frontmatter の `model` 指定は必須

スキル作成・更新時は frontmatter の `model` を **必ず明示する**。未指定だとセッション起動時のモデル（Opus が多い）がそのまま使われ、簡単な作業まで最上位モデルで走ってコストが膨らむ。

| モデル | 適するスキルの性質 |
|--------|------------------|
| `haiku` | 判断を伴わない定型的な抽出・置換・フォーマット変換のみ |
| `sonnet` | 機械的な差分検出・同期・整理・分類（技術判断を含まない） |
| `opus` | 複雑な推論・PDF/画像解析・多段の技術判断・文章生成 |

選定フロー：① 判断を一切伴わない定型処理 → `haiku`　② 機械的な同期・実行のみ → `sonnet`　③ 技術判断・推論・文章生成を含む（トラブルシューティング/設計判断/ナレッジ整理/ドキュメント同期/文章書き換え等） → `opus`。

迷ったら **`opus` を第一候補**にする（誤判断によるやり直し・コンテキスト浪費を防げるので、判断を伴うものは Opus が結局コスト効率良い）。明確に「機械的な同期・実行のみ」なら `sonnet`、極端な定型処理だけ `haiku`。

```yaml
---
name: <skill-name>
description: <発動条件が明確な説明>
argument-hint: [<引数ヒント>]
model: opus
---
```

## 既存スキル編集時のチェック

既存スキルを編集するついでに frontmatter を確認し：

- `model` が未指定なら**適切な値を補う**
- `model: sonnet` でも、技術判断・推論・文章生成を含むなら `opus` に**上げ直す**

ただしユーザーが明示的に別モデルを希望していればそれを優先。

## スキル設計時のコンテキスト配慮

スキルが大量出力（`git diff`・jq 結果・長いファイル群・ログ等）を扱うときの設計指針。大規模出力の一般的な扱い・subagent 委譲・並列起動の失敗パターンは [`development.md`](development.md) を参照。ここではスキル設計固有の点だけ：

- **SKILL.md は短く保つ**（100行以下目安。超えるなら別ファイルへ切り出し）
- **重い処理は `scripts/foo.sh` に切り出す**（シェルスクリプト内部の出力は親 context に流入しない）
- **長い subagent プロンプトは `prompts/foo.md` に切り出し**、起動時に `Read` で読み込む
- **規模判定すら親でやらない**（`git diff --stat` も出力肥大の元。最初から無条件で subagent / scripts に委譲）

| スキル種別 | 推奨設計 |
|----------|---------|
| 単純な情報取得・状態確認 | SKILL.md 内に bash 直接記述で OK |
| 大量出力を扱う（diff/log/jq） | scripts/ に切り出し、SKILL.md は呼ぶだけ |
| LLM 判断＋大量入力 | subagent 委譲、戻り値は構造化（JSON 等） |
| 複数言語/環境の同期処理 | scripts/ 内で完結（subagent 不要） |

> **設計だけでは context 限界は防げない**。長い会話＋MCP 大量ロード時はスキル本文の展開だけで超過しうる。重いスキル（`/commit-push`・`/sync-claude-code-settings` 等）は **`/clear` で新セッションにしてから呼ぶ**のが最終対策。
