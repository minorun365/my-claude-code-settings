# 技術ドキュメント執筆時の検証ルール

OSS への PR、技術ガイド、チュートリアル、Qiita 記事、チーム内ナレッジ、登壇資料の手順スライドなど、**「この手順で動きます」と他人に示すドキュメント**を書くときの検証ルール。

## 1. 通しテスト原則

手順書・チュートリアル系のドキュメントを書くときは、**自分の手で start から end まで一度通す**まで「動きます」「以下の手順でデプロイできます」と書かない。

通せない事情があるとき（AWS アカウント無し、有料サービス、本番環境必須、対象 OS が無い等）は、本文と PR description / 記事本文に **「未検証、要動作確認」と明示**する。「動くはず」「概ねこの手順で動く想定」のような曖昧な書き方で隠さない。

### 例外（通しテストを省ける場面）

- 既存の動作実績がある手順を**そのまま転載**するケース（公式 docs を改変せず引用）。ただしこの場合も「動作確認済みの引用です」と出典を明示
- ドキュメントの誤字修正やリンク切れ修正など、手順そのものに触らないケース

## 2. 公式ドキュメント vs 実装の乖離チェック

**新サービス（GA から 1 年以内 / Public Preview / Coming Soon ラベル付き）**の公式 docs を引用するときは、必ず以下を確認する：

1. **GitHub 実装側**で対応する機能が存在するか
   - 例：CLI のソースコード（`templates/`、`commands/` 等）に該当機能の実装ファイルがあるか
   - 例：SDK のリリースバージョンが docs に書かれた API を持つか
2. **GitHub Code Search** で docs 上のフラグ名・パラメータ名がリポジトリ内で参照されているか
3. lacking なら docs に **「公式ガイド記載に基づくが、実装側との整合性未確認」**と注記

### 情報源の信頼度ランキング

新サービス（特に AWS マネージドサービス）の最新仕様を確認するときは、以下の優先順位で一次情報を当たる：

| 優先度 | ソース | 例 |
|--------|--------|-----|
| **最優先** | **公式ベンダーの plugin / Connect / MCP が提供する一次情報ツール** | `mcp__plugin_aws-agents_awsknowledge__*`、AWS Agent Toolkit の `aws-core` / `aws-agents` skills |
| 高 | GitHub の `aws/` `awslabs/` 配下の公式リポの README・samples の最新コミット | `github.com/aws/agentcore-cli`、`github.com/awslabs/amazon-bedrock-agentcore-samples` |
| 中 | docs.aws.amazon.com・aws.github.io の公式チュートリアルページ | チュートリアル類は MCP ガイドより遅れて更新されるケースあり |
| 低 | tech-docs-searcher / WebFetch 経由の二次情報サマリ | 矛盾を含むまま返してくることがある |
| 参考のみ | AWS ブログ・Qiita 記事・サードパーティの解説 | 鮮度が落ちて deprecated 機能を案内しているケース多発 |

AWS 関連は raw `bedrock-agentcore-mcp-server` ではなく AWS Agent Toolkit plugin / awsknowledge を優先する。公式 plugin / Connect / MCP が使える場合は、WebFetch の二次情報サマリだけで済ませない。

## 3. 調査結果の矛盾を黙殺しない

サブエージェント（tech-docs-searcher 等）や WebFetch の戻り値に **「A と言うが、別ソースは B と言う」という矛盾**が含まれていたら、その時点で立ち止まる。片方を選んで進めない。

矛盾を発見した時の手順：

1. **どちらが新しいか・どちらが一次情報か**を明確にする（更新日付・MCP ガイド・GitHub release notes を確認）
2. それでも判断できなければ **公式 plugin / Connect / MCP の一次情報ツール** と **GitHub の最新コミット**で再確認
3. ユーザーへの報告でも矛盾点を明示し、「採用方針：X、理由：Y」と一言添える
4. **矛盾を黙殺して「片方を採用」する判断はしない**

## 4. 「判断を委ねた」≠「責任放棄」

みのるんに選択肢を示して指示をもらった場合でも、**実害が出る方針を指示された時は一段押し戻す**。

押し戻し例：

> 「動作未検証のまま発信するのはリスクが高いので、検証する時間を取らせてください」
>
> 「公式 docs と実装が乖離している件、影響範囲を確認したいです」
>
> 「ご指示は分かりましたが、提案 A だと L行 で失敗する可能性があります。提案 B を再検討しませんか？」

みのるんの指示は**最終決定ではなく、対話の起点**として扱う。指示通り進めて事故ったらみのるんの責任にはならず、Claude の判断責任。「OK 取った」を免罪符にしない。

## アンチパターン

- ❌ 公式 docs の手順を実機検証せずに技術ガイドとして書く
- ❌ 新サービスの公式 docs を「公式だから動く」と無条件に信用する
- ❌ みのるんから指示があったらリスクを伝えずにそのまま実行する
- ❌ 「ユーザーが OK と言ったから」と未検証のまま OSS に PR を出す
- ❌ docs / ガイド系の記事で「動作未検証」「未確認」を伏せる
- ❌ 「コピペしただろ？」と疑われるレベルの引用転載をオリジナル風に書く
- ❌ 中盤で気づいた実装乖離を、流れに押されてそのまま黙殺する
- ❌ **公式 plugin / Connect / MCP の一次情報ツールが使えるのに、tech-docs-searcher / WebFetch の二次情報サマリで済ませる**
- ❌ **サブエージェント戻り値の「A と言うが別ソースは B と言う」矛盾を黙殺して片方を採用する**（必ず一次情報で再確認）

## 適用タイミング

- OSS への PR を出す前
- 技術ブログ・Qiita 記事を公開する前
- チーム内ナレッジを共有する前
- スライド・登壇資料に技術的な手順を含める前
- みのるんへの「この手順でいけます」報告を出す前

迷ったら **「動作確認した？」「ソースの裏取りした？」と自問**してから発信する。
