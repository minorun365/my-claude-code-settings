# Subagents and edit tool pitfalls

## サブエージェントの報告に「判断」が含まれる場合は一次情報で確認する

### 背景

サブエージェント（`Agent` ツール）に調査を委任した場合、サブエージェントの報告はあくまで「解釈」であり、一次情報ではない。特に以下のような判断を含む報告は、**そのままユーザーに転送すると誤情報になりうる**。

- 「A と B は同じ箇所を指している（重複）」
- 「このコメントは不要」
- 「この差分は反映済み」

### 失敗パターン

PDF 校正注釈のチェックをサブエージェントで並行実行した際、あるサブエージェントが「p.2 の2つのレビューコメントが同一箇所に重複している」と報告した。実際には **2つの注釈は異なる箇所** を指していたが、「同じページ・同じトピック」という情報だけで「同一箇所」と誤判断していた。

### 対策

| 判断の重さ | 対処 |
|-----------|------|
| 漏れ・対応済みの確認（✅/❌一覧） | サブエージェントの報告をそのまま使っても問題が少ない |
| 「重複」「不要」「誤り」の指摘 | **自分で一次情報（PDFファイル等）を確認してからユーザーに伝える** |
| 削除・修正を伴うアクションの提案 | 必ず根拠を自分で確認してから提案する |

### PDFの注釈チェックの場合

注釈が「同じ箇所か」の判断は、Rect（座標情報）で客観的に確認できる。

```python
import fitz
doc = fitz.open("final_*.pdf")
for page in doc:
    for annot in page.annots():
        print(annot.rect, annot.info.get("title"), annot.info.get("content")[:50])
```

`annot.rect` の `y0` 座標が20pt以内であれば同じ行付近、`x0` が大きく異なれば別の列（左右ページ）。

## Edit ツール: ファイル Rename 後は Read 再実行が必須

`Bash mv old.md new.md` でファイル名を変更した直後に Edit ツールで `new.md` を編集しようとすると：

```
File has not been read yet. Read it first before writing to it.
```

Edit ツールは「直前に Read したファイルのみ Edit 可能」というキャッシュ判定をしており、**キャッシュはファイルパス単位**で管理されている。リネーム後の新パスは「未読扱い」になる。

### 対策

`mv` 後に必ず Read を挟む：

```bash
# 1. リネーム
mv meetings/2026-05-15.md meetings/2026-05-13.md
```

```
# 2. Read で再キャッシュ
Read meetings/2026-05-13.md
```

```
# 3. ようやく Edit 可能
Edit meetings/2026-05-13.md
```

### 余談: 全角括弧の罠も同時に踏みやすい

Read で出てくる日本語は全角括弧「（）」だが、`new_string` を半角「()」で書くとマッチしない。**Read 結果の `cat -n` 表示を見ながらコピペで old_string を書く**のが安全。
